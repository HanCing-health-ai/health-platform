## 一　背景與問題定義
### 1.1　從第一線看見的空白
2025 年，我開始在養生館擔任按摩師傅，同時進修傳統整復推拿課程。進入這個產業的第一天，我就發現一件讓我困惑的事：
每位師傅腦袋裡都裝著十幾年的臨床經驗，但這些經驗從來沒有被記錄下來。
每次服務前，師傅靠記憶回想客戶上次的狀況；每次交接，新師傅從零開始猜測。
整個產業的知識傳承，停留在師徒口耳相傳的階段。
我的同事處理一位久坐上班族的頸肩不適時，會先放鬆肩胛筋膜、再處理手臂，最後回到頸肩——這套順序比直接按頸肩的效果好得多。但這個判斷邏輯從未被文字化，下一位新進師傅完全無從得知。
身為正在學習的新手，我反而比資深師傅更清楚地感受到這個斷層：師傅的知識極其珍貴，但產業沒有任何工具幫助他們把知識轉化為可累積、可傳承、可驗證的資料。
這個觀察催生了 ConditionAI。我不是帶著十年經驗來做系統的老師傅——我是在學習過程中，看見了產業結構性空白的從業者，然後決定用 AI 工具鏈動手解決它。

### 1.2　產業現況觀察
台灣整復推拿相關登記業者達 4,800 家（衛福部民俗調理管理資料），從業人員超過 10 萬人，年產值逾 1,200 億元（經濟部統計處）。然而整個產業長期停留在師徒口傳模式，存在四大結構性問題：
*   **經驗傳承斷層**：高度依賴師徒口傳，技術無法文字化標準 → 品質不穩定，難以規模化
*   **資料模型缺失**：缺乏標準化客戶資料記錄格式 → 無法累積可分析的產業知識資產
*   **成效難以量化**：調理效果依賴主觀感受，無客觀指標 → 師傅無法比較方法有效性
*   **跨技術整合缺口**：整復、推拿、針灸資料完全不互通 → 客戶跨技術就診時資料斷鏈

### 1.3　全球市場趨勢驗證
*   **按摩服務全球需求急升**：ClassPass 2024：按摩預約量年增 39% → 核心服務市場擴大，數位化工具需求提升
*   **數位健康追蹤市場爆發**：2024 年 161 億美元，2034 年 680 億美元（CAGR 15.94%） → 亞太區最快成長，台灣具先行者優勢
*   **健康科技用戶年輕化**：57% 消費者使用健康科技，18-34 歲達 78-79% → 目標客群接受度高
*   **體感療癒主流化**：2026 全球趨勢報告：從另類療法進入主流 → 行為連動框架與主流認知對齊
*   **數據驅動 wellness**：數據驅動身體掃描服務年增 159% → 用戶對「看懂身體數據」的需求急升

### 1.4　兩個第一線洞察
**核心臨床觀察**：以下兩個洞察來自第一線服務的直接觀察，是整個系統設計的起點，也是最核心的競爭護城河。
*   **洞察一：主訴不等於根源**。上班族最常見的主訴是肩頸僵硬，但同一客戶往往同時伴隨手臂、手掌、腳底緊繃。調整服務順序——先處理肩胛筋膜再放鬆手部，回頭處理肩頸——效果明顯更好。這類跨部位的連動關係，憑主訴無法識別，唯有透過資料記錄與模式分析才能被驗證與傳承。
*   **洞察二：生活習慣是根本驅動力**。相同主訴的客戶，身體使用模式可能截然不同。久坐上班族的頸肩問題根源在靜態負荷累積；勞力工作者根源在重複性動態施力；長期低頭族則是特定肌群過度使用。所需的調理策略與生活調整方向各異。

**重要性說明**：如果系統只記錄主訴和調理部位，永遠無法建立「生活習慣與身體使用模式」的關聯資料庫。這個資料庫本身就是本系統最終的核心價值，也是任何競品在沒有第一線師傅知識的情況下無法複製的護城河。

### 1.5　焦慮即方向：系統設計的起點
**核心洞察**：
*   師傅每次服務前的焦慮：「這個客戶的狀況我要怎麼處理？上次做完有沒有效？」
*   工作室老闆的焦慮：「我不知道我的客戶為什麼不回來，也不知道師傅的調理方法有沒有用。」

ConditionAI 的起點就是這兩個焦慮——把焦慮變成產品，把不確定化成資料。
填完問卷立即獲得 AI 調理建議——這不是功能，而是把焦慮的發生時點（服務前 30 秒）變成系統的價值交付時點。目標是讓客戶在 3 分鐘內獲得「被理解」的感受，讓師傅在 30 秒內獲得有用的決策依據。師傅不是在「使用軟體」，而是在「獲得比自己記憶更可靠的判斷依據」。

### 1.6　亞健康市場定位
*   **醫療體系**：處理已病狀態 / 需要診斷與治療
*   **本系統 (亞健康族群)**：預防負荷累積 (外表健康但身體使用模式已累積問題) / 行為模式調整 (尚未達到就醫門檻)

**語言規範（本系統所有輸出嚴格遵守）**：
*   使用：「身體使用模式」「行為負荷」「調理建議」「行為調整建議」「舒緩方向」「改善趨勢」
*   禁止：「診斷」「治療」「疾病」「症狀」「風險」「藥物」等任何醫療用語

### 1.7　創辦人與團隊
**為什麼是我來做這件事**
第一，產品需求來自真實痛點。創辦人每天在第一線服務客戶，親身經歷「服務前不知道客戶上次做了什麼」「無法量化調理效果」的困境。ConditionAI 不是市場調查後設計的產品，而是從業者為了解決自己的問題而建造的工具。
第二，新手視角反而更敏銳。資深師傅已經習慣了紙本記錄和口頭交接，不覺得這是問題。身為正在學習的新進從業者，創辦人更能感受到「知識斷層」的痛感。
第三，AI 工具鏈從零學起的過程本身就是產品驗證。創辦人從零學習 Next.js、FastAPI、Supabase、Claude API，搭配 AI 開發工具，獨力完成了 V1 全功能系統的建置。這種「一人創辦人 + AI 工具鏈」的營運模式，正是 ConditionAI 主張的核心理念。

**團隊架構**
*   **V1 試點期**：創辦人 1 人 + AI 工具鏈 (現階段)
*   **V2 商業化**：＋兼職前端工程師 1 人、＋法律顧問（按件計酬） (10 家以上付費工作室)
*   **V3 平台化**：＋全職後端工程師 1 人、＋BD 1 人 (月營收穩定 NT$60,000 以上)

**核心說明**：單人團隊不是劣勢，而是刻意的策略選擇。在產品驗證階段，決策速度和迭代頻率比人力規模更重要。AI 工具鏈讓一個人完成過去需要六人團隊的產出——這本身就是 ConditionAI「用 AI 擴展人類能力」理念的最佳示範。

## 二　系統功能架構
### 2.1　雙向洞察引擎
ConditionAI 的核心架構是「雙向洞察引擎」——同一筆客戶輸入，產出兩個完全不同的輸出版本：
*   **師傅版（practitioner_output）**：行為模式分類 + 建議調理順序 + 重點部位 + 建議理由（可解釋式）。提供即時決策支援，縮短準備時間。
*   **客戶版（client_output）**：行為模式說明 + 生活調整建議（可執行小步驟）+ 次回建議時機。提升客戶自我認知，增強行為調整動機。

### 2.2　五層資料決策引擎
*   **L1 資料收集層**：標準化問卷 + 自然語言輸入 (Next.js 14 + Supabase questionnaire_responses)
*   **L2 安全過濾層**：Prompt Injection 掃描 + 風險分流規則模型 (FastAPI + triage_keywords.json)
*   **L3 AI 分析層**：Claude API Structured Output 雙向輸出 (actions.ts + validate_output() 詞彙過濾)
*   **L4 知識增強層**：知識庫 RAG 動態注入 (V1.5) (get_relevant_knowledge() 七層過濾 + Prompt 注入)
*   **L5 回饋學習層**：師傅採納率 + 客戶改善率追蹤 (treatment_records + follow_up_responses)

### 2.3　資料庫架構（Supabase 15 張資料表）
*   `studios` — 工作室主體，所有資料的頂層關聯 (V1)
*   `profiles` — 師傅資料，支援多技術標籤 specialty_tags[] (V1)
*   `clients` — 客戶基本資料，phone 唯一索引 (V1)
*   `questionnaire_responses` — 問卷填寫結果 (V1)
*   `triage_results` — 風險分流結果（A/B/C 類） (V1)
*   `behavior_tags` — AI 產出的行為模式標籤 (V1)
*   `follow_up_responses` — 定期回訪問卷資料 (V1)
*   `insight_reports` — AI 分析結果（師傅版 + 客戶版） (V1)
*   `treatment_records` — 調理紀錄（V1.5 D組新增 3 欄位） (V1)
*   `injection_attempts` — Prompt Injection 攻擊記錄 (V1)
*   `blocked_clients` — 被封鎖的客戶清單 (V1)
*   `body_sensation_logs` — 身體感受記錄 (V1)
*   `knowledge_base` — 知識庫條目（與 RAG 整合） (V1.5 B組)
*   `tag_library` — 標籤庫（知識條目分類） (V1.5 B組)
*   `credits` — 師傅知識貢獻點數 (V1.5 B組)
全部 15 張資料表均已啟用 Row Level Security（RLS）。知識三表已建立 GIN 索引以優化 RAG 查詢效能。

## 三　技術規格
### 3.1　技術棧總覽
前端：Next.js 14 + TypeScript + Tailwind CSS (Vercel) | 後端：Python FastAPI 3.11 (VPS) | 資料庫：Supabase（15 張表 + RLS） | AI 引擎：Claude API Structured Output | 工作流：n8n Cloud | 通知推送：LINE Messaging API | 版本控制：GitHub（master 分支）

### 3.2　工作流自動化（n8n Cloud）
*   工作流一：client_intake_v1 (V1 遺留)
*   工作流二：follow_up_trigger (V1 遺留)
*   工作流三：daily_summary (V1 遺留)
*   工作流四：knowledge-intake + review (✅ V1.5 B組 - Multi-Agent 知識審核流水線)

### 3.3　風險分流規則模型 V1
*   **C 類（暫緩調理）**：胸口心臟區域 / 頭部劇烈 / C類關鍵字庫 / 半年以上+下背+用藥中 → is_blocked = true，不產出建議，溫和提示
*   **B 類（加強監控）**：舊傷標記 / 用藥中 / 長期不適（3個月以上） → 繼續 AI 分析，加入黃色警示
*   **A 類（一般型）**：未觸發 C 或 B 條件 → 正常進入 Claude API 分析流程

### 3.4　資安強化五層防護
*   Layer 1: Webhook Header Authentication (V1 遺留)
*   Layer 2: Prompt Injection 前置掃描 ✅
*   Layer 3: LINE Messaging API Token 存環境變數 ✅ (已整合)
*   Layer 4: Supabase RLS 全表啟用 + 分級控制 ✅
*   Layer 5: VPS 強化（fail2ban + UFW + SSH Key Only） ✅ (或由未來建置)

### 3.5　知識庫架構與學術知識分層
知識庫來源分為六個領域，按可信度分層管理：
*   **中醫理論 / 物理治療**：高可信度，手動結構化 (V1)
*   **筋膜理論 / 解剖列車**：中高，需驗證手動 (V1.5)
*   **運動科學 / 人因工程學**：中，半自動 (V2)
*   **營養/飲食/睡眠**：低，僅作輸入變數 (V2 問卷擴充)

### 3.6　知識約束生成策略
*   **System Prompt 約束**：明確指示只能根據知識庫回答。
*   **RAG 強化注入**：七層過濾，僅注入高匹配度條目。
*   **信心分數連動**：低匹配度 → 低信心 + 保守建議。

### 3.7　合規過濾器（Sanitizer Agent）
任何文字只要會被客戶看到，就必須經過過濾：師傅專屬備註（V1.5 C/D組）、商品推薦理由（V2）。

### 3.8　知識管理 AI 專員系統（V1.5 B組已建置）
AI 專員 1 萃取 → AI 專員 2 標籤 → AI 專員 3 品質檢查 → 創辦人最終審核 → 狀態 active 並加 10 點 credits。

## 四　護城河與競爭策略
### 4.1　雙引擎飛輪模型
*   **知識飛輪（策略引擎）**：越多師傅使用 → 知識庫越豐富 → AI 輸出越精準 → 師傅越依賴。
*   **資料飛輪（成長引擎）**：客戶問卷 → 縱向資料 → 行為模式識別 → 效果可見化 → 工作室付費。

### 4.2　八條護城河
領域知識壁壘、縱向資料壁壘、切換成本、在地臨床知識、語言文化壁壘、工作流慣性與資料所有權、技術整合深度與可解釋性、社群知識。

### 4.3　知識品質量化指標
第一層（師傅採納率） × 第二層（客戶改善率）。V2 將引入學術里程碑（物理治療國考題、穴位考試界線）來定義邊界。

### 4.4 / 4.5 應對 Noterro 與 SaaS 崩塌
獨特的「時間差歷史追蹤」、在地術語、資料資產與 AI 深度綁定的邏輯，讓功能不再能被輕易複製。

## 六　商業模式與財務規劃
### 6.1　分階段商業模式
*   **第一階段 (0–6 月)**：Freemium（首 3 月免費）→ 基礎版訂閱。目標：10 家試點、500 筆資料、分流準確率 ≥85%。
*   **第二階段 (7–18 月)**：B2B + C端升級 + 導流 + 導購。目標：30 家付費、月營收 ≥NT$60,000。
*   **第三階段 (19–36 月)**：訂閱 + 課程 + ESG 報告。目標：200 家客戶、年營收 ≥NT$500 萬。

### 6.2　訂閱定價結構
**定價方向說明**：Freemium 設計：首 3 個月免費，無客戶數上限。時間限制比數量限制更能製造轉換壓力。
*   **永久免費版（師傅端）**：個人師傅體驗。每日 AI 分析次數限制、建議最多 3 點、1 筆知識庫注入。
*   **客戶端免費版**：客戶自助分析。行為模式分析 + 3 條生活調整建議 + 基礎趨勢。C 端獲客漏斗（V2 啟動）。
*   **客戶端升級版**：深度自助分析。完整分析 + 歷史趨勢 + 更多建議 + 推薦適合的師傅。
*   **專業方案（B2B）**：中型工作室。無限 AI 分析、完整知識庫、趨勢報告。
*   **企業方案（B2B）**：連鎖工作室。專業版全部 + 多方案生成 + B端後台 + ESG。

**C 端自助方案設計原則（V2）**：AI 輸出嚴格限制在「行為模式認知 + 生活調整小步驟」，不給任何按壓穴位或手法指導。有成效的客戶給予回饋 → 資料飛輪 → 推薦升級方案或推薦適合的師傅 → B 端獲客。

### 6.3　V1 月費估算
*   **Supabase**：NT$0（免費方案），500MB 足夠 V1 驗證。
*   **Claude API**：NT$500–2,000，每月 500–2,000 次。
*   **n8n Cloud**：NT$600–1,200，基礎方案。
*   **VPS（FastAPI）**：NT$200–300，2核2G。
*   **網域 / SSL**：NT$100。
*   **合計**：NT$1,400–3,600。V2 引入 Ollama 後 AI 成本降至趨近 0。

### 6.4　師傅填資料動機設計
即時價值（調理前 30 秒獲得洞察）、成就感（展示平均改善趨勢）、知識積累（回訪資料顯示有效性）、語音備注（轉文字）、知識貢獻點數（審核通過可用於費用抵扣）。

### 6.5　試點轉付費：轉換點設計
轉換點一：縱向資料的沉沒成本效應。轉換點二：試點期第 8 週的主動展示成效摘要。轉換點三：早鳥方案鎖定。

### 6.6　五層商業線地圖
*   **第一層：訂閱費（V1 啟動）**：工作室按月付費。
*   **第二層：預約導流 + 上帝視角儀表板（V2）**：抽成 5–15%。老闆看客群趨勢與師傅效能對比。
*   **第三層：商品導購（V2）**：零庫存聯盟模式，精準推薦。合規紅線：措辭限定「輔助日常行為調整的工具」。
*   **第四層：知識變現 + B2B2E 人才孵化（V2 起）**：內部虛擬督導，降低培訓成本。
*   **第五層：數據與認證 + 資料商業化（V3 以後）**：產業 benchmark 數據報告、ESG、RMN 概念。

### 6.7　財務模型框架
*   **第一年（V1）**：目標試點。主要收入為訂閱費。
*   **第二年（V2）**：目標 30 家，月營收 ≥NT$60,000。收入為 訂閱 + 導購佣金 + C端升級。
*   **第三年（V3）**：目標 200 家，月營收 ≥NT$420,000。收入為 訂閱 + 導購 + 課程 + ESG。

### 6.8　試點獲取策略
**外掛大腦定位**：談判核心話術：「我們是讓你的舊系統變聰明的外掛大腦，不是取代你舊系統的 ERP」。
**理想客戶輪廓（ICP）**：中小型工作室（老闆兼師傅最理想）、師傅對新工具開放、上班族為主的穩定客源。
**步驟**：創辦人工作室 → 同事 → 認識的老師 → 口碑卡片 → 第 8 週推送成效摘要。

### 6.9　知識冷啟動策略
*   **上線前**：30–50 條（創辦人審核）。
*   **試點期**：100–200 條（AI 專員預篩 + 人工審核）。
*   **V2 啟動前**：500+ 條（採納率 × 改善率雙重驗證）。

## 七　法規合規策略
### 7.1　台灣法規環境分析
依衛福部基準，傳統整復推拿不屬醫療行為。任何涉及「診斷」「治療」皆可能違法。系統所有 AI 輸出均經 `validate_output()` 詞彙過濾。需遵守個資法（告知義務、同意取得、安全維護、當事人權利）。

### 7.2　合規設計原則
語言規範落實、師傅最終決策權（`ai_adoption_level`）、風險分流阻斷（C 類主動阻斷）、知識審核人工介入、C 端輸出邊界。

**合規詞彙對照矩陣**：
*   禁止「治療/治好」，替換為「舒緩/調理」。
*   禁止「症狀/疾病」，替換為「身體使用模式/行為負荷」。
*   禁止「風險」，替換為「注意事項」。

### 7.3　各商業線法規檢查清單
*   **訂閱服務 (V1)**：AI 輸出語言風險。
*   **C 端自助 (V2)**：無師傅緩衝層，僅限生活調整建議。
*   **商品推薦 (V2 後期)**：排除藥妝，需上架審核流程。
*   **小兒分析 (V3 評估)**：兒童健康主張受更嚴格監管。

每年至少一次全面合規審查，啟動前諮詢法律顧問。
