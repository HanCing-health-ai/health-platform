# ConditionAI — 健康調理行為模式分析與決策支援引擎v2.7

> AI-powered behavioral pattern analysis and decision support engine  
> for traditional bodywork practitioners in Taiwan.

**系統定位：行為模式導向的健康管理工具，不涉及醫療診斷或治療**

---

## 專案簡介

台灣整復推拿產業逾 4,800 家業者、10 萬名從業人員，長期停留在師徒口傳模式，
缺乏系統化資料支撐。ConditionAI 透過五層資料決策引擎與雙 Agent 架構，
同時服務調理師與客戶：

- **調理師**：獲得跨部位連動分析與建議調理順序，精準定位問題根源
- **客戶**：獲得身體使用模式洞察與可行動的生活習慣調整建議

---

## 核心架構

### 五層資料決策引擎
| 層 | 名稱 | 功能 |
|---|---|---|
| 第一層 | 風險分流層 | A 一般型 / B 恢復期 / C 建議就醫，法規合規核心 |
| 第二層 | 背景標記層 | 職業負荷、舊傷史等多因素標記 |
| 第三層 | 行為分群層 | 多部位分布＋行為標籤→分群 |
| 第四層 | 可解釋式排序層 | 建議排序＋信心分數，非黑箱 |
| 第五層 | 驗證優化層 | 每日自動 review → 優化建議 → 人工決策執行 |

### 雙 Agent 架構
- **Agent A（分析引擎）**：接收問卷＋自然語言生活習慣描述，產出師傅版＋客戶版雙向洞察
- **Agent B（資料引擎）**：自動將結構化標籤寫入 Supabase，驅動分群邏輯更新

## 技術棧

| 工具 | 角色 |
|---|---|
| Next.js 14 | V1 正式前端介面 |
| Supabase | 資料庫，已完成 11 張資料表＋RLS 安全規則 |
| Python + FastAPI | 風險分流規則模型＋分群邏輯 |
| n8n | 自動化工作流（問卷→分流→寫庫→Line Notify） |
| Claude API | Agent A 初期推論（V2 換 Ollama 本地部署） |
| Langfuse | LLM 輸出監控（V2） |

---

## AI 輕量化設計

- **規則優先**：V1 以 Python 規則模型實作，無需 GPU、無需訓練資料
- **本地推論**：V2 引入 Ollama 7B 參數模型，資料不離境，成本趨近 0
- **漸進式演進**：規則分群 → K-means 聚類（資料 ≥500 筆後引入）
